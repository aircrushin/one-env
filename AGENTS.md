# Repository Guidelines

## Project Structure & Module Organization
- `src/routes/`: TanStack Router file-based routes (UI pages and API entrypoints like `api/v1/$.ts`).
- `src/lib/server/`: server-side domain logic (auth, repository access, versioning, service layer).
- `src/lib/shared/`: shared types/utilities used across client and server, with colocated unit tests.
- `src/components/`: reusable UI components.
- `public/`: static assets; `dist/`: production build output.
- `src/routeTree.gen.ts` is auto-generated by TanStack Router; do not edit manually.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: start local dev server on `http://localhost:3000`.
- `npm run build`: produce production bundle in `dist/`.
- `npm run preview`: serve the built app locally for verification.
- `npm run test`: run unit tests once with Vitest.

## Coding Style & Naming Conventions
- Language: TypeScript + React (ES modules, strict compiler settings).
- Match existing style: 2-space indentation, single quotes, no trailing semicolons.
- Component files use PascalCase (for example `Header.tsx`, `ThemeToggle.tsx`).
- Route files follow TanStack conventions (for example `__root.tsx`, `projects.$projectId.tsx`).
- Prefer path aliases `#/...` or `@/...` for imports from `src/`.

## Testing Guidelines
- Framework: Vitest (`vitest run`) with Testing Library and `jsdom`.
- Test files use `*.test.ts` and are colocated with the module under test.
- Add/adjust tests for behavior changes in `src/lib/shared` and `src/lib/server`, especially parsing, versioning, and API-adjacent logic.
- No enforced coverage threshold currently; cover new branches and edge cases.

## Commit & Pull Request Guidelines
- Current history is minimal (`init`), so use short, imperative commit subjects (preferably <= 72 chars).
- Keep commits focused and logically grouped; avoid mixing refactors with behavior changes.
- PRs should include: purpose summary, test evidence (`npm run test` output), linked issue/task, and screenshots for UI changes.

## Security & Configuration Tips
- Copy from `.env.example`; do not commit real secrets.
- Prefer `ONEENV_ADMIN_PASSWORD_HASH` over plain-text `ONEENV_ADMIN_PASSWORD`.
- `NOTION_API_TOKEN` and `NOTION_DATABASE_ID` are optional for local development (in-memory fallback is available).
